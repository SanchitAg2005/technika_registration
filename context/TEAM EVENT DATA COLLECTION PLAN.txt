		   TEAM EVENT DATA COLLECTION PLAN

=============================================================================================================================================================



1. TEAMS COLLECTION
=============================================================================================================================================================

Purpose:
Stores every team created for a team-based event.

Data Structure:

_id
teamId
eventId
leaderId
status
memberCount
createdAt
updatedAt

Field Description:

* _id           : MongoDB generated ObjectId.
* teamId        : Unique Team ID generated by the backend.
* eventId       : Event for which the team is created.
* leaderId      : Registration ID of the team leader.
* status        : forming | ready | registered | cancelled
* memberCount   : Current number of team members.
* createdAt     : Time when the team was created.
* updatedAt     : Time when the team was last modified.

=============================================================================================================================================================
2. TEAM MEMBERS COLLECTION
=============================================================================================================================================================

Purpose:
Stores every participant belonging to a team.

Data Structure:

_id
teamId
userId
role
joinedAt

Field Description:

* teamId     : Team to which the participant belongs.
* userId     : Registration ID of the participant.
* role       : Leader | Member
* joinedAt   : Time when the participant joined the team.

=============================================================================================================================================================
3. TEAM INVITATIONS COLLECTION
=============================================================================================================================================================

Purpose:
Stores every invitation sent by a team leader.

Data Structure:

_id
teamId
eventId
senderId
receiverId
status
createdAt
respondedAt

Status Values:

pending
accepted
declined
cancelled
expired

Field Description:

* teamId       : Team for which the invitation is sent.
* eventId      : Event associated with the invitation.
* senderId     : Registration ID of the team leader.
* receiverId   : Registration ID of the invited participant.
* status       : Current invitation status.
* createdAt    : Time when the invitation was created.
* respondedAt  : Time when the invitation was accepted or declined.

=============================================================================================================================================================
4. NOTIFICATIONS COLLECTION
=============================================================================================================================================================

Purpose:
Notifies users that an action requires their attention.

Data Structure:

_id
userId
type
referenceId
isRead
createdAt

Field Description:

* userId       : Participant receiving the notification.
* type         : TEAM_INVITE, INVITE_ACCEPTED, INVITE_DECLINED, etc.
* referenceId  : Invitation ID linked to this notification.
* isRead       : true / false
* createdAt    : Time when the notification was created.

Note:
Notifications should NOT be deleted after being accepted or declined.
Instead, mark them as "isRead = true". This preserves history while
using very little storage.

=============================================================================================================================================================
5. EVENT REGISTRATIONS COLLECTION
=============================================================================================================================================================

Purpose:
Represents the official registration of a participant in an event.

Data Structure:

_id
eventId
userId
teamId (nullable)
status
registeredAt

Individual Event Example:

eventId      : Chess
userId       : USER001
teamId       : null
status       : registered

Team Event Example:

eventId      : Cricket
userId       : USER001
teamId       : TEAM123
status       : registered

Note:
Every participant receives their own Event Registration record,
even when participating as part of a team.

=============================================================================================================================================================
VALIDATION RULES
=============================================================================================================================================================
1. A participant can register only once for a particular event.

2. A participant cannot belong to multiple teams for the same event.

3. A participant may belong to different teams for different events.

   Example:

   Tug of War  -> Leader: User A
   Cricket     -> Leader: User C

   This is completely valid because they are different events.

4. Invitations can only be sent to users who have already registered on the website.

5. A participant already registered in an event cannot receive another invitation for that same event.

6. The "Register Team" button becomes available only after the minimum required number of participants has joined.

7. Every accepted invitation automatically creates:

   * A Team Member record.
   * An Event Registration record.

8. MongoDB is the Source of Truth.
   Google Sheets are synchronized only for organizer reporting and management.
